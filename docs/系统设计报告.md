# 学生宿舍信息管理系统设计报告

## 1. 总体架构

系统采用前后端分离思想：

- 表现层：`src/main/resources/web` 中的 HTML、CSS、JavaScript 构建企业后台界面。
- 接口层：`DormitoryWebServer` 提供登录、学生住宿管理、调换申请、审批和统计分析 REST 接口。
- 业务层：`StudentDormService`、`ChangeRequestService` 负责核心业务规则。
- 数据层：`MysqlStudentRepository`、`MysqlChangeRequestRepository` 负责 MySQL 数据读写。
- 分析层：`DormAnalysisService` 负责调用大模型接口或本地规则分析。

## 2. 核心类说明

| 类名 | 说明 |
| --- | --- |
| `DormitoryManagementSystem` | 程序入口，默认启动 Web 服务 |
| `DormitoryWebServer` | HTTP 服务和 REST API |
| `MysqlDatabaseInitializer` | 自动创建数据库、数据表和演示数据 |
| `MysqlStudentRepository` | 学生住宿信息 MySQL 仓储 |
| `MysqlChangeRequestRepository` | 调换申请 MySQL 仓储 |
| `StudentDormRecord` | 学生宿舍基本信息实体 |
| `DormChangeRequest` | 宿舍调换申请实体 |
| `StudentDormService` | 添加、修改、删除、查询、排序、统计和宿舍更新 |
| `ChangeRequestService` | 提交申请、查看申请、同意或拒绝申请 |
| `DormAnalysisService` | 统一智能分析入口 |

## 3. 数据库设计

### 3.1 students 表

| 字段 | 含义 |
| --- | --- |
| student_id | 学号，主键 |
| name | 姓名 |
| department | 所在系 |
| class_name | 班级 |
| dorm_number | 宿舍号 |
| dorm_phone | 宿舍电话 |
| bed_number | 床位 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

约束：`student_id` 唯一；`dorm_number + bed_number` 唯一，避免同一床位重复入住。

### 3.2 change_requests 表

| 字段 | 含义 |
| --- | --- |
| id | 申请编号，主键 |
| student_id | 学号 |
| current_dorm_number | 原宿舍 |
| current_bed_number | 原床位 |
| target_dorm_number | 目标宿舍 |
| target_dorm_phone | 目标宿舍电话 |
| target_bed_number | 目标床位 |
| reason | 调换理由 |
| status | 审核状态 |
| created_at | 申请时间 |
| handled_at | 审核时间 |
| admin_comment | 审核意见 |

## 4. 关键流程

### 4.1 添加和修改学生住宿信息

管理员在 Web 表单中录入信息，系统校验必填字段、学号是否重复、床位是否占用。校验通过后写入 MySQL。

### 4.2 删除学生住宿信息

管理员在表格中选择记录删除。系统根据宿舍号和学号共同定位记录，避免误删。

### 4.3 宿舍调换申请

普通用户提交目标宿舍、目标床位和理由。管理员审核时如果选择同意，系统会再次检查目标床位是否可用，然后自动更新学生住宿表，并把申请状态改为已同意。

### 4.4 智能宿舍分析

管理员选择楼栋或宿舍后，系统统计入住人数、床位容量、空余床位和各系入住比例。若配置大模型接口，系统调用接口生成自然语言建议；若未配置，则使用本地规则生成可演示建议。
